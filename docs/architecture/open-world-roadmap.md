# Open-world technical roadmap

この文書は、`novel-game-prototype` をブラウザ上の3D探索型ノベルRPGとして段階的に育てるための技術ロードマップです。

参考方針は、WebGL/Three.js系の軽量オープンワールド事例から得られる次の考え方です。

- 巨大な汎用ゲームエンジンではなく、Three.js + 必要最小限の補助ライブラリで作る
- 3D探索はWebGL、会話・メニュー・ノベルUIはDOM/CSSに分離する
- 高品質化はポリゴン量だけでなく、アート方針、ライティング、ポストプロセス、ベイクで作る
- 大容量モデルや音声をGitHubリポジトリに詰め込みすぎない
- 将来的な衝突判定は、重い物理エンジンよりも空間分割・BVHを検討する

---

## 1. 現在の基本方針

現時点では、王都アウレリアを平面ベースの城郭都市として育てる。

短期的には、球体惑星型ワールドや本格BVH衝突判定へ飛ばず、まず以下を完成度高くする。

- 現在の王都マップの探索性
- クエスト導線
- ミニマップ・目的地表示
- 進行ガード
- 馬車襲撃イベントのアクション化
- 主要施設の別マップ化
- NPCと馬車の破綻軽減

球体世界、カスタム重力、G-Bufferアウトライン、BVHは中長期の研究項目とする。

---

## 2. レイヤードアーキテクチャ

このプロジェクトでは、3D空間に直接すべてのUIを描くのではなく、次の2層構成を維持する。

| レイヤー | 担当 | 内容 |
|---|---|---|
| WebGL layer | Three.js / `game_world.js` | 3D空間、プレイヤー移動、カメラ、火球、NPC、マップ生成 |
| DOM UI layer | HTML / CSS / small JS | 会話、選択肢、HUD、ミニマップ、メニュー、進行ガード |

### 理由

- DOMは日本語テキスト、折り返し、アクセシビリティ、CSS装飾に強い
- Three.jsは3D描画と空間表現に集中できる
- 会話UIやメニューをWebGL内で再実装しないため、開発負荷を抑えられる
- `pointer-events` と body class により、探索中・会話中・メニュー中の操作権限を切り替えやすい

## 3. 短期ロードマップ

### Phase 1: プレイ導線の安定化

優先度: 高

- ミニマップと次目的地表示
- 目的地マーカー
- Mキー拡大地図
- 進行ガード
- 条件未達時の分かりやすい通知
- クエスト順序の破綻防止

目的は、「何をすればいいか分からない」「順番を飛ばせる」を減らすこと。

### Phase 2: 馬車襲撃のアクション化

優先度: 高

現状の荷車襲撃を、選択肢だけでなく火球アクションで進行させる。

理想フロー:

```text
森の街道
→ 荷車襲撃現場
→ モンスター出現
→ Jキーで火球
→ 命中で商人救助
→ 紹介状入手
→ 北門通過
```

実装方針:

- 既存の火球処理を再利用する
- 専用の複雑な戦闘AIはまだ入れない
- まずは1体の簡易モンスターと1発命中でよい
- 進行ガードと連携させる

### Phase 3: 主要施設の別マップ化

優先度: 中

- 魔法学院キャンパス
- 教会大聖堂・庭園区画
- 訓練場の拡張
- 宿屋・商店街の内部強化

王都全体を一気に巨大化するより、重要施設を別マップ化して密度を上げる。

---

## 4. 中期ロードマップ

### NPC・馬車のすり抜け改善

現在は簡易colliderと移動ロジックで成立しているが、NPCや馬車が建物や屋台を貫通する可能性がある。

中期的には、以下の順で進める。

1. 移動先がblockedなら進まない
2. blockedなら方向転換する
3. 目的地を再選択する
4. 主要な屋台・馬車・大型小物にもcolliderを付ける
5. 必要なら軽量な空間分割を強化する

### 固定NPCのT字ポーズ改善

外部glTF NPCの導入後、T字ポーズが残る場合がある。

対応案:

- 目立つ固定NPCだけ、Blenderで腕を下げた待機ポーズGLBを作る
- 全NPCにAnimationMixerを入れるのは後回し
- 通行人は軽量性を優先する

## 5. 長期ロードマップ

### three-mesh-bvhの検討

将来的に、建物・地形・階段・坂・複雑な屋内形状が増えた場合、現在のAABB中心の簡易colliderだけでは限界が来る。

その段階で `three-mesh-bvh` を検討する。

導入目的:

- 大きな環境メッシュへの高速レイキャスト
- 地面判定
- 段差・斜面・壁面スライド
- 複雑な地形の衝突判定

ただし、すぐには入れない。

理由:

- 現状の王都はプリミティブ＋AABBでかなり制御できる
- BVH導入はキャラクターコントローラー全体に影響する
- 実装ミス時の破壊範囲が大きい

### 手描き風アウトライン・ポストプロセス

長期的には、アニメ調の見た目を強化するためにアウトラインやポストプロセスを検討する。

候補:

- 深度ベースのエッジ検出
- 法線ベースのエッジ検出
- 軽いSobel filter
- 低解像度レンダー + アップスケール
- ノイズによる線の揺らぎ

ただし、モバイルや低スペック環境では重くなりやすいため、`?render=high` 限定機能として検討する。

### 球体世界は別プロトタイプで実験

球体惑星型の世界は魅力的だが、現在の王都アウレリアとは設計が大きく異なる。

必要になるもの:

- カスタム重力
- 毎フレームのup vector更新
- quaternionによる姿勢制御
- 専用カメラ
- 地表接地判定
- 既存王都座標系の再設計

そのため、本編へ直接入れず、将来 `experiments/spherical-world/` のような別プロトタイプで検証する。

---

## 6. 実装ルール

このプロジェクトでは、今後も以下を守る。

- 直接mainへpushしない
- 新機能は1テーマ1PR
- `game_world.js` を触るPRでは変更範囲を明示する
- 大きい機能ほどdocsを先に書く
- UI補助は可能なら別JS/CSSに分離する
- 既存の会話・進行・マップ生成を壊さない
- 低画質モード `?render=low` を必ず意識する
